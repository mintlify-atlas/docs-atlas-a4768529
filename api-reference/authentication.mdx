---
title: 'Authentication'
description: 'How to authenticate as an agent or a provider node, and how to pass optional agent identity headers.'
---

The OAHL API uses two distinct bearer token schemes — one for agents and one for provider nodes. Every request must include an `Authorization` header.

## Agent authentication

Agent-facing endpoints (`/v1/capabilities`, `/v1/requests`, `/v1/sessions/*`) require an **agent API key**.

```bash
Authorization: Bearer <agent_api_key>
```

Agent API keys grant access to device discovery, session management, and capability execution. Access to individual devices is further controlled by each device's [access policy](/api-reference/capabilities#device-access-policy).

## Provider authentication

Provider-facing endpoints (`/v1/provider/*` and `/ws/provider`) require a **provider API key**.

```bash
Authorization: Bearer <provider_api_key>
```

Provider API keys are used by hardware nodes to register themselves, poll for commands, and submit results.

## Which endpoints require which token

| Endpoint group | Required token |
|---|---|
| `GET /v1/capabilities` | Agent API key |
| `POST /v1/requests` | Agent API key |
| `POST /v1/sessions/{id}/execute` | Agent API key |
| `POST /v1/sessions/{id}/execute-batch` | Agent API key |
| `POST /v1/sessions/{id}/stop` | Agent API key |
| `POST /v1/provider/nodes/register` | Provider API key |
| `GET /v1/provider/nodes/{id}/poll` | Provider API key |
| `POST /v1/provider/nodes/results` | Provider API key |
| `WSS /ws/provider` | Provider API key |

## Optional agent identity headers

Agent endpoints accept three optional headers that enable per-device access policy enforcement. You do not need to include these headers if the device has `public` visibility.

<ParamField header="x-agent-id" type="string">
  Stable identifier for the calling agent or its owner. Used for `allowed_agents` / `denied_agents` policy checks.
</ParamField>

<ParamField header="x-agent-org-id" type="string">
  Organisation the agent belongs to. Used for `allowed_orgs` policy checks.
</ParamField>

<ParamField header="x-agent-roles" type="string">
  Comma-separated list of roles the agent holds (e.g. `admin,read-only`).
</ParamField>

<Warning>
  In v1, these headers are **self-asserted**. They are not cryptographically bound to the bearer token. Do not rely on them as a security boundary in high-trust environments.
</Warning>

## Example request with all auth headers

```bash
curl https://oahl.onrender.com/v1/capabilities \
  -H "Authorization: Bearer <agent_api_key>" \
  -H "x-agent-id: agent-abc123" \
  -H "x-agent-org-id: org-xyz" \
  -H "x-agent-roles: operator"
```
