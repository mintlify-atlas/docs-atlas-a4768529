An adapter is a device-specific plugin that translates OAHL's standard capability model into the protocol and API calls required by a piece of physical hardware. The adapter is the only component in OAHL that ever touches hardware directly — everything above it (the node, the session manager, the policy engine, the Cloud Registry) speaks the standard `Adapter` interface defined in `@oahl/core`.

When your node starts, it loads each package name listed in the `plugins` array of `oahl-config.json`, constructs the exported adapter class, and calls `initialize()`. From that point on, the node routes validated execution requests to the right adapter based on the `adapter` field on each device entry.

## How adapters fit into OAHL

```
Agent  →  Cloud Registry  →  Node  →  Adapter  →  Hardware
                           (policy)  (translate)  (physical device)
```

The node never speaks hardware protocols directly. It calls `execute(deviceId, capabilityName, args)` on the adapter and receives an `ExecutionResult`. The adapter decides how to turn that call into an ADB command, a V4L2 ioctl, an RTL-SDR tune operation, or anything else the hardware needs.

## Installing an adapter

Use `oahl install` to fetch a published adapter from npm and automatically register it in your `oahl-config.json`:

```bash
oahl install @oahl/adapter-usb-camera
```

The command runs `npm install` under the hood and appends the package name to the `plugins` array if it is not already present. Restart the node to activate the adapter:

```bash
oahl start
```

To remove an adapter:

```bash
oahl remove @oahl/adapter-usb-camera
```

### Installing a local adapter

If you are developing an adapter locally or have the source checked out, install it by path:

```bash
npm install ./path/to/your/adapter
```

Then add the package name (the `name` field from the adapter's `package.json`) to `plugins` manually:

```json
{
  "plugins": [
    "@oahl/adapter-mock",
    "my-local-adapter-name"
  ]
}
```

## Loading adapters from oahl-config.json

The `plugins` array in `oahl-config.json` controls which adapters the node loads at startup:

```json
{
  "node_id": "local-node-01",
  "cloud_url": "https://oahl.onrender.com",
  "provider_api_key": "123456",
  "provider": {
    "name": "Local Lab"
  },
  "plugins": [
    "@oahl/adapter-mock",
    "@oahl/adapter-usb-camera"
  ],
  "devices": [
    {
      "id": "usb-cam-1",
      "type": "camera",
      "adapter": "usb-camera-adapter",
      "capabilities": ["capture_image"],
      "policy": {
        "public": false,
        "max_session_minutes": 20
      }
    }
  ]
}
```

The node resolves each plugin name using Node.js `require()`. Every package in `plugins` must be installed in `node_modules` before starting the node.

<Note>
  The `adapter` field on a device entry must match the `id` string returned by the adapter class (e.g. `'usb-camera-adapter'`), not the npm package name. Check the adapter's source or README to confirm the correct value.
</Note>

## Using the adapter registry

The OAHL registry at `registry/adapters.txt` lists all officially tracked adapter packages. The registry crawler reads each package's `oahl-manifest.json` to populate the [OAHL Marketplace](https://oahl.org/marketplace). The full registry index is available at `registry/registry.json`.

Currently tracked adapters:

```text
@oahl/adapter-usb-camera
@oahl/adapter-rtl-sdr
@oahl/adapter-mock
@oahl/adapter-android
```

## Building a custom adapter

If no first-party adapter covers your hardware, you can build your own. Any npm package that exports a class implementing the `Adapter` interface works as an OAHL plugin.
